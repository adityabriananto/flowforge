# FlowForge

> **FlowForge is an Engineering Workflow Platform.**
> **It separates Planning, Execution, and Review while preserving Engineering Knowledge through the Engineering Workspace.**

FlowForge has been successfully validated through real-world engineering projects (including *BroDev Meeting Intelligence* and *BroDev Cashier*). The workflows and architecture documented here reflect the production workflow currently used by the FlowForge maintainers.

---

## 1. Philosophy

FlowForge does not replace developers. FlowForge does not replace AI.

FlowForge orchestrates Engineering State. 

Developers remain responsible for engineering decisions. AI Providers execute engineering work. 

FlowForge specifically manages:
- **Engineering Workspace**
- **Mission Lifecycle**
- **Planning Context**
- **Runtime**
- **Engineering State**

## 2. Human in the Loop

FlowForge intentionally keeps developers in control at all times. FlowForge never makes engineering decisions autonomously. 

Developers are responsible for:
- Defining engineering goals
- Approving missions
- Selecting execution providers
- Reviewing engineering outputs
- Completing missions

## 3. Validated Engineering Workflow

The engineering workflow has been exercised end-to-end using multiple AI providers. This workflow separates the planning phase from the execution phase, ensuring that the architecture remains provider-agnostic. 

You can mix and match AI providers based on their strengths (e.g., using *ChatGPT* as the Planning Provider, *Codex* as the Execution Provider, and *Claude* as the Review Provider).

```text
Developer
   ↓
Planning AI
   ↓
Mission
   ↓
Compile
   ↓
Mission Package
   ↓
Execution AI
   ↓
Engineering Deliverables
   ↓
Review AI
   ↓
Mission Complete
```

## 4. Engineering Identity

FlowForge utilizes a dual-identity model to ensure the system is human-friendly while remaining technically robust.

- **Engineering Identity** (e.g., `PROJECT-005`): The canonical, human-facing identifier. This is used by Developers, Documentation, the CLI, and AI Providers for communication.
- **System Identity** (e.g., `8b4b4b8d-...`): The internal UUID. Used strictly internally by the Runtime, Persistence layer, and Repository Integrity. Developers should rarely, if ever, need to interact with UUIDs.

## 5. Workspace Architecture

FlowForge maintains strict separation between your long-term engineering knowledge and transient execution state.

### Engineering Workspace (`engineering/`)
This is the persistent, long-term memory of your project. It contains:
- Missions
- Deliverables
- `PROJECT_STATE.yaml`
- `WORKSPACE.yaml`
- Engineering Knowledge (RFCs, ADRs)

### Runtime Workspace (`.flowforge/`)
This is the transient state managed entirely by FlowForge during execution. It contains:
- Mission Packages
- Sessions
- Runtime Metadata
- Cache

*(Note: Mission Packages are considered runtime artifacts, not long-term Engineering Knowledge).*

## 6. Mission Packages

```text
Mission
   ↓
Compile
   ↓
PROJECT-005.package.yaml
   ↓
Execution Provider
```

Mission Packages are vendor-agnostic runtime artifacts generated by the Mission Compiler. They combine the declarative Mission intent with the project's Engineering Context, producing deterministic instructions ready to be handed to an AI Execution Provider.

## 7. Quick Start

Ensure you have **`uv`** installed, then execute your first mission:

```bash
# 1. Initialize the Engineering Workspace
flowforge init

# 2. Author a new Mission
flowforge mission new

# 3. Compile the Mission into a Package
flowforge compile PROJECT-000

# 4. Start and Execute the Mission
flowforge mission start PROJECT-000
flowforge run PROJECT-000

# 5. Review deliverables and Complete the Mission
flowforge mission complete PROJECT-000
```

## 8. CLI Reference

FlowForge provides a streamlined command-line interface for managing the entire lifecycle.

- `flowforge init`: Initialize a workspace.
- `flowforge doctor`: Diagnose environment and provider health.
- `flowforge mission new`: Launch the authoring wizard.
- `flowforge compile <ID>`: Compile a mission package.
- `flowforge run <ID>`: Execute a compiled package.

For all commands and options, consult the [Detailed CLI Reference](docs/cli.md).

## 9. Provider Architecture

FlowForge is strictly provider-independent. AI execution engines are configured declaratively in a `providers.yaml` file, ensuring your workspace never gets locked into a specific vendor's SDK or API.
