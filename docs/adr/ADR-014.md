# ADR-014: Long-Term Memory and Context Restoration via Engineering State

## Status
Accepted

## Context
FlowForge memerlukan kemampuan bagi AI provider yang berbeda untuk berkolaborasi dan melanjutkan pengerjaan proyek lintas sesi tanpa harus bergantung pada riwayat percakapan chat LLM atau log database runtime. Proyek harus memiliki memori rekayasa jangka panjang yang independen dan tersimpan langsung di dalam Engineering Workspace.

## Decision
Kami memutuskan untuk mengimplementasikan **Engineering State** sebagai satu-satunya sumber kebenaran (*single source of truth*) siklus hidup rekayasa proyek:
1.  **Domain Model**: Memetakan status proyek, status workspace, status misi, history provider, knowledge, keputusan, blockers, rekomendasi, dan timeline secara terstruktur dalam dataclass Python.
2.  **Abstraksi Port & Adapter**: Mengikuti Clean Architecture dengan meluncurkan `EngineeringStateRepository` untuk mengisolasi persistensi.
3.  **Persistensi YAML**: Menyimpan seluruh status secara idempotent di bawah berkas `engineering/PROJECT_STATE.yaml` dalam Engineering Workspace.
4.  **State Service**: Menyediakan `EngineeringStateService` untuk memanipulasi transisi dan context recovery melalui method `resume_engineering_state`.

## Consequences
-   **Kelebihan**: AI provider berikutnya dapat langsung memulihkan seluruh konteks proyek dari status terakhir yang tertulis di disk tanpa perlu melakukan pemindaian ulang seluruh codebase dari nol atau memuat riwayat obrolan AI yang panjang.
-   **Kelebihan**: State rekayasa dapat dimasukkan ke dalam version control (git) bersama kode sumber proyek.
-   **Kekurangan**: Validasi schema YAML ekstra harus dijalankan setiap kali memuat file untuk menghindari modifikasi manual yang merusak data.
