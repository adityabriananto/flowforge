# ADR-006: Mission-Driven Workflow Engineering System

## Status
Accepted (Hardened - v1)

## Context
FlowForge didesain sebagai framework otonom yang mengorkestrasi Human, AI, dan System Workers. Selama ini, pemicu utama (source of truth) masih diwakili oleh workflow dan database internal yang secara tidak langsung terikat pada parameter runtime.

Kami membutuhkan sebuah konsep abstrak di atas workflow yang merepresentasikan "tujuan bisnis" atau "tujuan rekayasa" (business or engineering goals) yang independen dari implementasi runtime vendor AI. Konsep ini kami sebut sebagai **Mission System**.

Dalam iterasi hardening (FF-012.1), kami menetapkan standarisasi skema v1 untuk menjamin stabilitas data masukan.

## Decision
1. **Mission Domain Model**: Kami mengimplementasikan Mission domain model sebagai dataclass murni yang berisi identitas, deskripsi, goals, status, dan metadata extensibel.
2. **Siklus Hidup (Lifecycle) & MissionState Enum**: Misi mendukung transisi eksklusif menggunakan `MissionState` Enum: `BACKLOG`, `READY`, `ACTIVE`, `REVIEW`, `DONE`, `ARCHIVED`. Ini menggantikan string literal mentah untuk ketik-aman (type safety).
3. **YAML-First & Schema Validation**: Sumber kebenaran eksternal untuk Mission dimuat dari berkas YAML menggunakan `MissionLoader`. Loader memvalidasi tipe skema (version 1), format UUID, tingkat prioritas (`low`, `medium`, `high`, `critical`), field wajib, dan melarang ID misi yang duplikat.
4. **Decoupled Ports & Adapters**: Menyediakan `MissionRepository` sebagai port abstrak untuk pemisahan data persistent dengan implementasi konkret in-memory (dan DB di masa depan).

## Consequences
- FlowForge sekarang dapat digerakkan secara fungsional berdasarkan pencapaian misi (Mission System).
- Data Misi terisolasi sepenuhnya dari logic pemrosesan prompt dan API vendor LLM.
- Validasi ketat mencegah terjadinya kesalahan parsing format masukan di awal runtime (*fail-fast*).
