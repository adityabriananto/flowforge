# ADR-015: Traceable Orchestration and Immutability via Engineering Session

## Status
Accepted

## Context
Selama pengerjaan orkestrasi otomatis oleh worker AI, kita memerlukan log jejak terperinci (*audit log*) dari setiap aktivitas rekayasa tunggal. Data eksekusi ini harus berdiri sendiri secara terpisah agar worker berikutnya dapat memahami persis apa yang terjadi, apa saja berkas yang berubah, keputusan arsitektur yang diambil, dan status sisa tugas, tanpa perlu bergantung pada riwayat percakapan chat. Sesi ini harus bersifat permanen (*immutable*) setelah selesai untuk menjamin auditabilitas.

## Decision
Kami memutuskan untuk menerapkan **Engineering Session** sebagai log eksekusi formal:
1.  **Domain Model**: `EngineeringSession` melacak metadata sesi (started_at, duration, status: `CREATED`, `ACTIVE`, `COMPLETED`, `FAILED`, `CANCELLED`), ringkasan rekayasa, berkas yang berubah, decisions, blockers, recommendations, dan draf serah terima (*handover*).
2.  **Immutability**: Log sesi bersifat immutable (tidak dapat dimodifikasi) begitu status transisi memasuki kondisi final (`COMPLETED`, `FAILED`, `CANCELLED`).
3.  **Persistensi Terpisah**: File log sesi ditulis ke `.flowforge/logs/session_<id>.yaml` untuk memisahkan berkas log runtime dari berkas rekayasa yang dapat diedit langsung oleh manusia di folder `engineering/`.
4.  **State Integration**: Menyediakan kait data (*integration points*) yang menyuplai `EngineeringStateService` untuk memperbarui status proyek jangka panjang secara akumulatif setelah sesi rekayasa sukses diselesaikan.

## Consequences
-   **Kelebihan**: Transparansi eksekusi yang luar biasa tinggi. Jejak rekayasa model AI terdokumentasi rapi.
-   **Kelebihan**: Asersi imutabilitas mencegah kerusakan data log historis secara tidak sengaja oleh proses berikutnya.
-   **Kekurangan**: Folder `.flowforge/logs/` akan menyimpan banyak file sesi seiring berjalannya proyek, sehingga membutuhkan kebijakan pembersihan cache log di masa mendatang.
