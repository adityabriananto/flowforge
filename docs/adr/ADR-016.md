# ADR-016: Provider Interface and Runtime Registry

## Status
Accepted

## Context
AI Runtime FlowForge harus menjamin independensi total dari model AI spesifik (Ollama lokal, API Claude, API Gemini, dsb.). Logic orkestrasi internal runtime tidak boleh memiliki kopling atau pengasumsian tentang cara pemanggilan spesifik vendor AI mana pun. 

## Decision
Kami memutuskan untuk mengimplementasikan antarmuka abstraksi provider baru:
1.  **AIProvider Port**: Port abstrak minimal di `ports/ai_provider.py` dengan method `name()`, `health()`, dan `execute(mission_package, engineering_state)`.
2.  **AIRuntimeProviderRegistry**: Registry runtime dinamis di `services/runtime/provider_registry.py` untuk mengelola daur hidup dan pencarian instansi adapter provider.
3.  **ProviderConfigLoader**: Parser konfigurasi di `services/runtime/provider_config_loader.py` untuk memuat provider terdaftar dari konfigurasi berkas YAML eksternal (`providers.yaml`).

## Consequences
-   **Kelebihan**: Independensi model AI mutlak. Integrasi AI baru hanya memerlukan penambahan file adapter plugin tanpa menyentuh core engine FlowForge.
-   **Kelebihan**: Transparan mendukung provider cloud-based API maupun CLI-based lokal (Ollama).
-   **Kekurangan**: Modul orkestrasi runtime di masa depan harus mematuhi interface `AIProvider` secara ketat untuk mempertahankan batas domain arsitektural.
