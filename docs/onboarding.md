# Developer Onboarding Guide

Welcome to FlowForge! This guide will walk you through the FlowForge Engineering Workflow in about ten minutes. By the end, you'll understand not just how to run commands, but *why* FlowForge exists and how it structures AI-assisted software engineering.

---

## 1. The Core Philosophy

Before typing any commands, it is crucial to understand **why FlowForge separates planning from execution.**

When you use a traditional AI chat assistant to write code, the conversation is volatile. If you close the chat, the AI loses all context about why a database table was designed a certain way, or why a specific architecture pattern was chosen.

FlowForge solves this by introducing the **Engineering Workspace**. FlowForge acts as the operating system for your project's repository. Instead of chatting, you manage **Missions** (engineering tasks). The knowledge gained from completing a mission is permanently written to your repository's **Engineering State**. 

**The Golden Rule:**
> FlowForge plans engineering. Providers execute engineering. Developers approve engineering decisions.

---

## 2. Setting Up the Engineering Workspace

### How
Navigate to your project directory and run:
```bash
flowforge init
```

### Why
This command initializes the repository. FlowForge detects your framework (e.g., Laravel, React, Django) and sets up the `engineering/` directory. This folder becomes the permanent home for your project's memory (`PROJECT_STATE.yaml`), architecture decisions (`adrs/`), and mission files (`missions/`).

You have now established a **Planning Context**—a unified snapshot of your project's current state that the AI will read before writing any code.

---

## 3. Mission Authoring

### How
You need a new feature. Let's create an engineering task:
```bash
flowforge mission new
```
The Interactive Mission Wizard will prompt you for:
1. **Mission Title** (e.g., "Implement User Authentication")
2. **Business Goal** (e.g., "Allow users to log in securely using JWT tokens")
3. **Priority** (High/Medium/Low)

### Why
FlowForge forces you to declare intent. Instead of writing a complex, hacky prompt telling the AI *how* to code, you simply define the *business objective*. The Planning Engine automatically translates this into a structured Mission Draft YAML inside `engineering/missions/backlog/`.

---

## 4. The Mission Compiler

### How
Now that the mission is in the backlog, we need to prepare it for the AI:
```bash
flowforge compile PROJECT-000
```

### Why
The **Mission Compiler** merges your brief business goal with the rich context of your Engineering Workspace (the framework, the existing architecture, the project state). It produces a **Mission Package**—a vendor-neutral, heavily detailed set of instructions. This ensures that the AI doesn't just write generic code, but code that perfectly fits *your* specific repository.

---

## 5. Activating and Executing the Mission

### How
Move the mission out of the backlog:
```bash
flowforge mission start PROJECT-000
```
Then, execute it:
```bash
flowforge run PROJECT-000
```

### Why
When you hit `run`, the **Runtime Engine** takes over. It is entirely stateless and does not know how to generate code on its own. Instead, it delegates the Mission Package to your active **Provider** (e.g., Claude, Ollama, OpenAI). 

Because FlowForge uses a Provider abstraction, your engineering process never gets locked into a single AI vendor. The provider executes the tasks and writes the code.

As the provider works, an immutable **Engineering Session** log is recorded. If something goes wrong, you have a complete, step-by-step audit trail of exactly what the AI did.

---

## 6. Completing the Mission

### How
Review the code changes generated by the AI. If the tests pass and you approve of the work:
```bash
flowforge mission complete PROJECT-000
```

### Why
This is the most critical step. Completing the mission updates your `PROJECT_STATE.yaml` (the **Engineering State**). FlowForge records the new capabilities that were just added to the system. 

When you start your *next* mission, FlowForge will read this updated state. The next AI agent will instantly know about the authentication system you just built, without you having to explain it in a chat prompt.

---

## Summary

You have just completed the full lifecycle:
1. You established a permanent memory (**Engineering Workspace**).
2. You declared an objective (**Mission Authoring**).
3. FlowForge translated it into context (**Mission Compiler** & **Mission Package**).
4. An AI vendor wrote the code (**Runtime** & **Provider**).
5. The knowledge was saved permanently (**Engineering State**).

You are now ready to build scalable, AI-assisted software using FlowForge!
